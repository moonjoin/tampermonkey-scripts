# 摘要结果操作按钮可配置设计

日期：2026-07-11
状态：已确认方案，待实现

## 1. 现状

当前结果区由 `.tabbit-result-actions` 承载操作按钮，但按钮不是配置驱动的：

- 正常摘要在 `finalizeSummaryUI()` 中直接创建“复制摘要、编辑摘要、生成配图、复制生图提示词、摘要发评论、HTML PPT、发送 FLOMO、下载字幕、下载 SRT”等按钮。
- 生图结果在 `showImageResult()` 中再次直接创建一套文字结果按钮，并在图片卡片内创建“保存图片、填字并点上传”按钮。
- 兜底模式、HTML PPT 直出模式等路径也会单独组装下载按钮。
- SRT 按钮只有在 `hasStructuredSubtitleData(rawSubtitleBody)` 为真时才出现；没有可用时间轴时不会显示。
- 设置统一使用 `CONFIG` 和 `bili_summary_pro_config` localStorage 配置，`loadConfig()` 通过 `Object.assign(DEFAULT_CONFIG, saved)` 兼容缺失字段。
- 保存设置后会更新模型、预设等部分页面状态，但结果操作按钮没有统一的“按配置重新渲染”入口。

## 2. 范围

### 2.1 本次包含

1. 为截图中的摘要结果操作按钮增加显示/隐藏配置。
2. 在设置面板使用复选框控制显示，并提供“上移”“下移”调整顺序。
3. 正常摘要结果和生图结果共用同一份按钮配置、同一套顺序规则。
4. 保存设置后立即刷新当前已显示的结果按钮，不重新请求摘要或图片。
5. 没有时间轴字幕时跳过 SRT 按钮。
6. 允许用户关闭全部按钮。
7. 旧配置没有新字段时默认全部显示；旧配置不能因为新增字段而失效。

### 2.2 不包含

- 不改按钮本身的业务行为、文案、样式或 API 调用逻辑。
- 不改变图片自动下载开关；“保存图片”按钮的显示配置与自动下载是两回事。
- 不为评论区、弹幕分析、全面分析、聊天输入区等非摘要结果操作区增加配置。
- 不把按钮配置写入 `POSITION_KEY`，也不改变面板位置配置。

## 3. 操作按钮目录

使用稳定的内部 ID，不以中文文案作为配置键。默认顺序沿用当前正常摘要结果区的主要顺序；图片专属按钮仍按其所属结果区域渲染。

| ID | 默认文案 | 适用结果 | 说明 |
| --- | --- | --- | --- |
| `copy_summary` | 📋 复制摘要 / 📋 复制文字 | 正常摘要、生图结果 | 复制当前结果文字 |
| `edit_summary` | ✏️ 编辑摘要 | 正常摘要、生图结果 | 编辑当前结果文字 |
| `generate_image` | 🖼️ 生成配图 / 发送到 Flow | 正常摘要、生图结果 | 手动生成或重新生成配图；Flow 模式沿用现有文案 |
| `copy_image_prompt` | 复制 生图提示词 | 正常摘要、生图结果 | 复制生图提示词及当前摘要 |
| `post_comment` | 📋 摘要发评论 | 正常摘要、生图结果 | 将摘要填入 B 站评论框 |
| `html_ppt` | 📊 HTML PPT | 正常摘要、生图结果 | 使用当前摘要生成 HTML PPT |
| `send_flomo` | 发送 FLOMO | 正常摘要、生图结果 | 发送当前摘要到 Flomo |
| `download_transcript` | 💾 下载字幕 | 正常摘要、生图结果、兜底结果 | 下载当前字幕文件 |
| `download_srt` | 🕒 下载 SRT | 正常摘要、生图结果、兜底结果 | 仅在存在有效时间轴字幕时显示 |
| `save_image` | 💾 保存图片 | 已生成图片的生图结果 | 下载当前图片；只在图片操作行显示 |
| `fill_image_comment` | 💬 填字并点上传 | 已生成图片的生图结果 | 图片评论辅助操作；只在图片操作行显示 |

“模型名”和编辑状态提示不是操作按钮，不进入可配置列表；它们继续作为结果区的状态信息显示。按钮适用但当前不可执行时，沿用现有 disabled/loading 行为，不因配置改变。

## 4. 数据结构

在 `DEFAULT_CONFIG` 增加：

```js
resultActionButtons: [
  { id: 'copy_summary', enabled: true },
  { id: 'edit_summary', enabled: true },
  { id: 'generate_image', enabled: true },
  { id: 'copy_image_prompt', enabled: true },
  { id: 'post_comment', enabled: true },
  { id: 'html_ppt', enabled: true },
  { id: 'send_flomo', enabled: true },
  { id: 'download_transcript', enabled: true },
  { id: 'download_srt', enabled: true },
  { id: 'save_image', enabled: true },
  { id: 'fill_image_comment', enabled: true }
]
```

约束：

- 数组顺序就是渲染顺序。
- `enabled: false` 是合法值；不能用“空数组”代表未配置，因为空数组必须表示用户关闭全部按钮。
- 每个 ID 只能出现一次。
- 渲染层只接受已知 ID；未知 ID 不渲染，也不阻塞其他配置。
- 配置项只保存 `id` 和 `enabled`，按钮文案、适用场景和事件绑定由代码中的按钮目录维护。

建议提供纯函数 `normalizeResultActionButtons(raw)`：

1. 字段不存在、不是数组或无法解析时，返回默认全显示列表。
2. 字段存在且为数组时，保留其中已知 ID 的顺序和 enabled 值。
3. 对缺失的已知 ID 追加到末尾，并设为 `enabled: true`，保证升级后新增按钮默认出现。
4. 去重时保留第一次出现的记录。
5. 对 `enabled` 做布尔归一化；只有明确为 `false` 时关闭，其余有效条目默认开启。
6. 输入为 `[]` 时返回空数组，不自动补回按钮。

这样既能兼容没有 `resultActionButtons` 的旧配置，也能保留“全部关闭”的用户选择。

## 5. UI 设计

在现有设置面板中增加一个可折叠设置组，例如标题“🎛️ 摘要结果操作按钮”。

每行包含：

- 复选框：控制该按钮是否显示。
- 按钮名称：显示稳定的用户可理解文案。
- 上移：非首项可点击；点击后与上一项交换。
- 下移：非末项可点击；点击后与下一项交换。

交互规则：

- 顺序调整只修改设置面板内的编辑副本，不立即改变结果区。
- 复选框修改只修改编辑副本；允许取消所有复选框。
- 上移/下移后立即更新列表 DOM，并保持当前行可见；不触发摘要请求。
- 保存按钮按当前列表整体写回 `CONFIG.resultActionButtons`。
- 恢复默认设置时，按钮列表恢复为默认顺序且全部勾选。
- 结果区没有任何启用按钮时，保留 `.tabbit-result-actions` 容器为空，不显示占位提示，不留下多余间距。

设置行的按钮文案应与结果区实际文案对应；Flow 模式、图片结果等动态文案不影响其稳定 ID。

## 6. 渲染流程

统一抽出按钮配置读取和渲染职责，避免在正常摘要、生图、兜底路径分别维护一套判断：

```text
loadConfig()
  -> normalizeResultActionButtons()
  -> CONFIG.resultActionButtons

结果生成或重新渲染
  -> renderResultActionButtons(contentDiv, context)
       -> 按配置顺序遍历
       -> 过滤 enabled=false
       -> 过滤当前 context 不适用的按钮
       -> 对 download_srt 再检查 hasStructuredSubtitleData(rawSubtitleBody)
       -> 创建按钮并绑定现有事件
```

`context` 至少包含：

- `resultType`：`summary`、`image`、`fallback` 或 `html_ppt`。
- `hasText`：当前是否有可复制/编辑的文字结果。
- `hasImage`：当前是否有成功生成的图片。
- `hasTimelineSubtitle`：当前是否存在可生成 SRT 的时间轴。
- `videoInfo` 和 `contentDiv`：复用现有事件处理所需上下文。

渲染要求：

- 正常摘要的 `finalizeSummaryUI()` 改为调用统一渲染器。
- 生图结果的文字按钮区改为调用同一渲染器；图片卡片的 `save_image`、`fill_image_comment` 也从同一配置中决定是否显示。
- `showApiNotConfiguredFallback()` 和 HTML PPT 直出等仍可传不同 context，但不得绕过配置直接创建结果操作按钮。
- `appendSubtitleDownloadButtons()` 不再直接决定“下载字幕/SRT”是否出现，改为提供下载动作或由统一渲染器调用；SRT 的时间轴过滤集中在渲染器。
- 每次结果重置、摘要完成、图片生成完成、编辑完成后，都可以安全地重复调用渲染器；渲染器应先清空目标按钮区，避免重复按钮和重复事件。
- 保存配置后的刷新只重绘当前页面已有结果，不重新执行 `runSummary()`、不清理摘要缓存、不重新生成图片。

顺序边界：统一列表控制同一结果区域内的按钮顺序。图片专属按钮只在图片操作行中参与相对排序；当某个按钮因当前结果不适用或 SRT 无时间轴而被过滤时，其他按钮按配置中的相对顺序紧密排列。

## 7. 兼容与异常处理

### 7.1 旧配置

- 旧 localStorage 没有 `resultActionButtons`：加载后使用默认列表，全部显示。
- 旧配置只有部分新 ID：按保存顺序显示已有项，缺失项追加并默认显示。
- 旧配置出现未知 ID、重复 ID、非法 enabled 值：按数据结构中的归一化规则处理，不影响 API、预设和其他配置。
- 不主动改写旧配置；只有用户点击保存时才把归一化后的新字段持久化，避免启动时产生无意义写入。

### 7.2 SRT

- `download_srt` 已启用但 `hasStructuredSubtitleData(rawSubtitleBody)` 为假：不渲染按钮。
- 时间轴数据为空、字段不完整或 `buildSrtContent()` 不能生成内容：同样跳过按钮。
- 其他字幕下载按钮不受 SRT 可用性影响。
- 若用户点击保存后字幕时间轴状态发生变化，刷新时根据最新状态重新判断。

### 7.3 全部关闭

- `resultActionButtons: []` 或所有条目 `enabled: false` 都表示用户明确关闭全部按钮。
- 不自动补回默认按钮，不显示“至少保留一个”错误。
- 结果内容、图片、状态提示和其他非结果操作仍正常显示。

### 7.4 保存和刷新失败

- localStorage 写入失败时沿用现有 `saveConfig()` 错误处理；不要先刷新再假设保存成功。
- 保存成功后先更新内存 `CONFIG`，再对当前面板中的 `.tabbit-result-actions` 和图片操作行执行刷新。
- 如果当前没有结果、面板已关闭或目标节点不存在，保存仍成功，后续结果生成时使用新配置。
- 刷新过程不得因单个按钮事件绑定异常而阻塞其他按钮；必要时记录 warning，并保留已有错误处理风格。

## 8. 验收标准

### 配置与兼容

- [ ] 新安装或没有新字段的旧配置，11 个按钮全部按默认顺序显示；SRT 仍受时间轴条件控制。
- [ ] 读取旧配置不会覆盖 API、预设、生图等已有字段。
- [ ] 缺失、重复、未知或非法按钮配置不会导致设置面板打不开或摘要结果不渲染。
- [ ] 空列表和全部取消勾选都能保存，重新打开设置后仍保持全部关闭。

### 设置交互

- [ ] 设置面板逐项显示复选框、按钮名称、上移和下移。
- [ ] 首项不能继续上移，末项不能继续下移；交换顺序后列表展示立即正确。
- [ ] 保存后 localStorage 中保存稳定 ID、enabled 状态和当前顺序。
- [ ] 恢复默认后顺序和勾选状态恢复为默认值。

### 结果刷新

- [ ] 保存后当前已显示的正常摘要按钮立即按新配置刷新，不重新请求 AI。
- [ ] 保存后当前已显示的生图结果文字按钮和图片操作按钮立即按同一配置刷新。
- [ ] 后续重新生成摘要、重新生成图片、切换预设或进入兜底模式时仍使用同一配置。
- [ ] 重复刷新不会产生重复按钮、重复点击事件或旧按钮残留。
- [ ] 全部关闭时结果操作容器为空且布局无明显空白。

### 条件显示

- [ ] 没有有效时间轴时，即使 `download_srt` 勾选，也不显示 SRT 按钮。
- [ ] 有有效时间轴且 `download_srt` 勾选时显示并能正常下载。
- [ ] 关闭某个按钮只影响显示，不改变其他按钮的原有行为。
- [ ] 图片未生成时不显示图片专属“保存图片”和“填字并点上传”；图片生成后按配置显示。

## 9. 自检结论

- 文档中没有遗留未完成占位标记。
- “全部关闭”与“旧配置默认全显示”已区分：缺失字段使用默认值，显式空列表表示用户关闭全部。
- “共用配置”已限定为统一按钮目录、稳定 ID、顺序和 enabled 状态；按钮是否适用当前结果仍由运行时上下文判断。
- SRT 的“勾选”和“有无时间轴”是两层条件，二者没有冲突。
- 保存后刷新只重绘现有 DOM，不重新请求或生成结果，符合立即刷新且不产生额外成本的要求。
